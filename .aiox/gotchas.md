### Job Status column vs Real-time computation divergence

**[WARNING]**

The "status" column in the jobs table is a static snapshot and can become stale during active processing.

**Workaround:** Always use JobStatusService.computeJobStatus() for real-time accuracy. Use the sync logic (shouldSyncStatus) to update the DB column upon completion to maintain listing performance.

**Related Files:** backend/src/services/jobStatusService.ts, backend/src/controllers/jobController.ts, backend/src/controllers/statusController.ts

### Docker Volume Permission (EACCES) with Non-root Users

**[CRITICAL]**

When mounting host directories as volumes, the application may fail to write (EACCES) if running as a non-root user defined via 'USER' in the Dockerfile, as it cannot modify permissions of the mounted volume.

**Workaround:** 1. Install 'su-exec' in the Dockerfile. 2. Remove the fixed 'USER' command. 3. Use an ENTRYPOINT script that starts as root, performs 'chown -R user:group /path', and then executes the app command using 'su-exec user ...' to drop privileges.

**Related Files:** backend/Dockerfile, backend/docker-entrypoint.sh, docker-compose.yml

## Runtime

### Campo Quill vazio pode enviar HTML vazio como <p><br></p>

**[WARNING]**

Quando o usuário deixa o campo Quill vazio (sem digitar nada), o valor do estado é '<p><br></p>' em vez de string vazia. Isso pode causar comportamento inesperado no backend ou emails com corpo vazio mas não completamente vazio.

**Workaround:** No hook useEmailSubmit ou no componente EmailForm, adicionar validação: if (!body || body === '<p><br></p>') return null/empty. O backend também deve validar html antes de enviar.

**Related Files:** frontend/src/components/EmailForm/EmailForm.tsx, frontend/src/hooks/useEmailSubmit.ts

### Quill JS cleanup pode ter edge cases em componentes aninhados

**[WARNING]**

 Quill.js faz cleanup automático ao desmontar o componente, mas em casos de componentes aninhados ou re-renders rápidos, o editor pode não ser removido corretamente causando memory leaks ou erros de 'Quill is not defined'.

**Workaround:** Garantir que o componente QuillEditor use React.useEffect cleanup return no useEffect para chamar quill?.destroy(), OU verificar que react-quill faz isso automaticamente (verificar documentação da versão instalada).

**Related Files:** frontend/src/components/QuillEditor/QuillEditor.tsx, frontend/src/components/EmailForm/EmailForm.tsx

## Security

### HTML malicioso em user input - sanitização no backend

**[CRITICAL]**

O HTML enviado pelo usuário através do QuillEditor pode conter scripts maliciosos (XSS), iframes, ou código berbahaya. O frontend confia no input do usuário e o backend deve sanitizar antes de enviar via SMTP.

**Workaround:** No backend, usar biblioteca como 'sanitize-html' ou 'dompurify' para limpar tags dangerous: script, iframe, object, embed, link (com javascript:), event handlers (onclick, onerror, etc). whitelist de tags seguras: p, br, strong, b, em, i, u, a (somente http/https), ul, ol, li, h1-h6, blockquote, pre, code.

**Related Files:** backend/src/services/emailService.ts, backend/src/controllers/emailController.ts

## Test

### E2E Tests devem usar .ql-editor selector ao invés de textarea para Quill

**[WARNING]**

Ao substituir textarea por QuillEditor, testes E2E que usam locator('textarea') vão falhar porque Quill usa .ql-editor como elemento editável.

**Workaround:** Atualizar todos os Selectors de testes de 'textarea' para '.ql-editor' com playwright locator: page.locator('.ql-editor').fill('texto')
