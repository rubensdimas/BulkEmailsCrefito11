## 📋 User Story

**Como** desenvolvedor/usuário do sistema,
**Quero** que as permissões de escrita no diretório de uploads estejam corretamente configuradas nos containers,
**Para** que o upload de arquivos de contatos (XLSX) funcione sem erros de "EACCES: permission denied".

---

## 🎯 Objective

Corrigir o erro de permissão `EACCES` que ocorre quando o backend tenta salvar arquivos na pasta `/app/uploads`. O problema reside na discrepância entre o usuário que executa o processo dentro do container e o dono do volume montado no host.

---

## 🎯 Acceptance Criteria

```gherkin
GIVEN os containers estão em execução
WHEN o usuário envia um arquivo para o endpoint /api/upload
THEN o arquivo deve ser salvo com sucesso na pasta /app/uploads
AND a API deve retornar status 200/201 com os detalhes do upload

GIVEN a pasta de uploads no host não tem permissões de escrita para o usuário do container
WHEN o container inicia
THEN o entrypoint deve corrigir as permissões automaticamente (se executado como root inicialmente)
OR o docker-compose deve mapear o usuário corretamente
```

## 📝 Dev Notes

### Key Files
- `backend/Dockerfile`: Define o usuário `nodejs`
- `docker-compose.yml`: Define os volumes e comandos de inicialização
- `backend/docker-entrypoint.sh`: Script de inicialização que deve tratar as permissões

### Technical Notes
- Em modo development, o `docker-compose.yml` mapeia `./backend:/app`, o que faz com que o comando `command` no compose (`npm run dev`) ignore o `USER nodejs` sem configuração explícita.
- O erro `EACCES` geralmente ocorre porque a pasta `uploads` no host pertence ao usuário local, e o container tenta escrever nela com um usuário que não tem os mesmos IDs.

- **Test file location**: N/A (Manual/Infra validation)
- **Test standards**: Docker healthchecks and manual I/O verification
- **Testing frameworks**: `docker compose logs`, `ls -la` checks
- **Specific requirements**: Validate that the entrypoint script correctly handles existing files in the volume.

- O `docker-entrypoint.sh` foi alterado para iniciar como `root`, corrigir o `chown` da pasta de uploads e então fazer o drop de privilégios para `nodejs` via `su-exec`.
- O Dockerfile foi ajustado para instalar o `su-exec` e remover o `USER nodejs` fixo, delegando a troca de usuário ao entrypoint.
